# payment-system

基于微服务的支付系统设计与实现。

## 项目简介

本项目面向支付场景，基于微服务架构拆分账户、交易、对账、用户等核心模块，重点实现转账主流程、消息驱动对账、异常状态流转与补偿机制。项目主要用于训练后端工程能力，并为后续简历投递、毕业设计和性能优化研究提供实践基础。

## 技术栈

- Java
- Spring Boot
- MySQL
- Redis
- RocketMQ
- Seata
- Maven
- Docker Compose

## 模块划分

- `payment-account`  
  账户模块，负责账户信息、余额变更、充值、扣款、流水记录等能力。

- `payment-trade`  
  交易模块，负责转账主流程、交易记录生成、交易成功消息发送等能力。

- `payment-reconcile`  
  对账模块，负责消费交易成功消息、生成对账记录、查询统计、异常处理与补偿任务。

- `payment-user`  
  用户模块，提供用户相关基础能力。

- `payment-common`  
  公共模块，封装通用返回结构、DTO 等基础组件。

## 当前已实现功能

### 1. 微服务基础拆分
- 完成账户、交易、对账、用户、公共模块的项目拆分
- 支持各模块独立启动与协作

### 2. 转账主流程
- 支持转账请求发起
- 支持账户余额扣减与增加
- 支持交易记录落库
- 支持成功与失败交易结果区分

### 3. 消息驱动对账
- 交易成功后发送 RocketMQ 消息
- 对账模块异步消费交易成功消息
- 自动生成对账记录
- 实现交易与对账解耦

### 4. 对账查询能力
- 查询全部对账记录
- 按交易号查询对账记录
- 按状态筛选对账记录
- 对账汇总统计查询

### 5. 对账异常处理
- 支持单条异常标记
- 支持单条异常恢复
- 支持批量异常补偿
- 支持定时自动补偿任务

### 6. 基础幂等控制
- 已异常记录不可重复标记
- 非异常记录不可恢复
- 无异常记录时补偿任务可正确返回

### 7. 接口与工程优化
- 将状态修改类接口由 GET 重构为 POST
- 优化统一返回结构
- 支持业务成功/失败分流及失败原因提示

### 8. 基础性能验证
- 已对对账查询类接口完成第一轮基础并发压测
- 初步验证系统在中低并发场景下的可用性

## 核心业务流程

### 转账与对账主链路
1. 用户发起转账请求
2. 交易模块执行转账主流程
3. 账户模块完成余额变更
4. 交易模块记录交易结果
5. 交易成功后发送 RocketMQ 消息
6. 对账模块消费消息并生成对账记录
7. 若出现异常，可通过异常标记、恢复、批量补偿、定时补偿完成修复

## 对账状态说明

- `0`：待对账
- `1`：对账成功
- `2`：对账异常

当前支持的状态流转：
- `0 -> 1` 正常对账成功
- `1 -> 2` 手动标记异常
- `2 -> 1` 手动恢复
- `2 -> 1` 批量补偿恢复
- `2 -> 1` 定时任务自动补偿恢复

## 项目亮点

- 基于 RocketMQ 实现交易与对账解耦，体现最终一致性处理思路
- 对账模块形成异常标记、恢复、批量补偿、定时补偿的完整闭环
- 基于状态校验实现基础幂等控制，避免重复操作导致数据状态错乱
- 持续优化接口设计与返回结构，增强联调效率和异常排查能力
- 已完成基础压测，为后续分页优化和性能分析提供依据
