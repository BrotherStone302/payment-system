# payment-system

基于微服务架构设计与实现的支付系统项目，围绕账户管理、转账交易、消息驱动对账、异常处理与补偿机制等核心场景展开，用于训练 Java 后端工程能力，并沉淀可用于简历展示和面试讲解的项目实践。

---

## 1. 项目背景

支付类系统通常具有交易链路长、状态流转复杂、数据一致性要求高等特点。相比单体项目，基于微服务拆分后的支付场景更能体现服务协作、异常处理、幂等控制、消息解耦和补偿机制等后端核心能力。

本项目以“用户转账”场景为主线，拆分账户、交易、对账、用户、公共模块，完成了从**转账请求发起、交易处理、账户余额变更、消息通知、对账落库，到异常标记、恢复与补偿**的一条完整业务闭环。

---

## 2. 项目目标

本项目的主要目标包括：

- 搭建一个具备基础业务闭环的微服务支付系统
- 训练 Spring Boot 微服务项目拆分与模块协作能力
- 实践支付场景下的交易状态管理、异常处理与补偿机制
- 基于 RocketMQ 实现交易与对账模块的异步解耦
- 沉淀一套可验证、可展示、可讲解的后端项目成果

## 4. 系统模块

### 4.1 payment-account
账户模块，负责账户信息管理与资金变更相关能力，包括：

- 账户开户
- 账户余额查询
- 扣款与入账
- 账户流水记录
- 按用户查询账户及流水信息

### 4.2 payment-trade
交易模块，负责转账主流程编排，是系统的核心业务模块，包括：

- 接收转账请求
- 校验交易参数
- 校验 tradeNo 幂等性
- 调用账户模块完成扣款与入账
- 记录交易单状态
- 发送交易成功消息

### 4.3 payment-reconcile
对账模块，负责交易结果的异步核对与异常处理，包括：

- 消费交易成功消息
- 生成对账记录
- 对账记录分页查询与状态筛选
- 异常标记与恢复
- 手动补偿与定时补偿

### 4.4 payment-user
用户模块，提供用户相关基础能力，为账户、交易等模块提供用户维度支撑。

### 4.5 payment-common
公共模块，封装项目通用组件，包括：

- 统一返回结构 `Result`
- 通用 DTO
- 公共异常处理
- 公共基础类

## 5. 核心业务流程

### 5.1 转账主流程

系统当前已经完成如下主链路：

1. 用户发起转账请求
2. 交易模块校验请求参数与交易单号
3. 交易模块创建交易记录
4. 账户模块完成转出账户扣款
5. 账户模块完成转入账户入账
6. 交易模块更新交易结果
7. 交易成功后发送 RocketMQ 消息
8. 对账模块消费消息并生成对账记录
9. 对账成功后记录最终状态

对应业务闭环如下：

**转账请求 → 交易单创建 → 扣款 / 入账 → 账户流水记录 → 发送成功消息 → 对账服务消费消息 → 对账记录生成 → 对账成功**

## 7. 失败场景与异常处理

为了提升项目完整度与可讲解性，当前已围绕核心转账链路设计并验证多类失败场景，包括：

- 重复 `tradeNo` 提交
- 转账金额为 0
- 转账金额为负数
- 转出账户不存在
- 转入账户不存在
- 转出用户与转入用户相同
- 余额不足
- 缺少必填字段
- 查询不存在交易单

对于异常场景，系统支持：

- 返回明确失败原因
- 保证交易状态可追踪
- 对异常记录进行标记、恢复和补偿
- 通过账户流水、交易单、对账记录三类数据进行链路验证

## 8. 对账状态说明

当前对账状态定义如下：

- `0`：待对账
- `1`：对账成功
- `2`：对账异常

已实现的状态流转包括：

- `0 -> 1`：正常对账成功
- `1 -> 2`：手动标记异常
- `2 -> 1`：手动恢复
- `2 -> 1`：手动补偿恢复
- `2 -> 1`：定时任务自动补偿恢复

## 9. 项目亮点

### 9.1 基于消息驱动实现交易与对账解耦
交易成功后通过 RocketMQ 发送事件消息，由对账模块异步消费并生成对账记录，体现了基于消息机制实现模块解耦和最终一致性处理的设计思路。

### 9.2 具备完整的异常处理闭环
项目不仅实现了成功主流程，还针对支付场景下的典型失败情况设计了异常标记、手动恢复、批量补偿和定时补偿机制，提升了系统完整度。

### 9.3 强调状态流转与幂等控制
围绕交易单号、对账状态、异常记录等关键对象进行状态控制，避免重复提交和非法状态切换导致的数据混乱问题。

### 9.4 形成可验证的业务链路
通过交易单、账户流水、对账记录三类数据，可以完整追踪一次转账请求从发起到落库、从成功到对账、从异常到补偿的全过程。

### 9.5 具备一定工程表达能力
项目已沉淀 `.http` 测试文件、分页查询接口、失败场景测试、统一返回结构和模块化代码组织，具备进一步优化为简历项目的基础。
